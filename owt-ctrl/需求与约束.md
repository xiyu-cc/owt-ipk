# 需求与约束

## 1. 项目目的

- 系统应当为个人站点提供“远程控制内网设备”的能力，使操作者可以在公网安全入口下对家庭/实验室局域网设备执行开机、重启、关机与运行参数管理。
- 系统应当服务于两类主要使用者：
  - 站点操作者（通过浏览器进行控制与状态查看）。
  - 网关设备维护者（在网关侧部署并维护执行代理）。
- 系统预期价值：
  - 在不暴露内网设备入站端口的前提下完成远程控制。
  - 将“控制决策”与“设备执行”分离，降低公网暴露面。
  - 提供可追溯的命令与操作记录，便于回溯与排障。

## 2. 范围

### 2.1 本项目包含

- 提供远程控制能力：开机、重启、关机、状态采集开关、参数读取、参数保存。
- 提供设备在线状态与采集状态的实时推送能力。
- 提供命令执行全生命周期查询能力（提交、状态变化、结果查询）。
- 提供按设备维度的控制参数持久化能力。
- 提供访问控制与操作审计能力。
- 提供面向网关设备与控制服务的长连接控制通道。

### 2.2 Out of Scope（不包含）

- 不包含局域网设备的入站暴露与直连控制方案。
- 不包含多租户账号体系与复杂 RBAC 权限模型。
- 不包含浏览器直接连接网关设备。
- 不包含控制服务本机执行设备动作（设备动作仅由网关执行）。
- 当前阶段不包含旧版本协议/旧行为兼容与双栈灰度。

## 3. 核心对象与术语

- **控制服务**：面向公网入口的控制与状态汇聚系统，负责命令编排、状态管理、查询与审计。
- **网关执行代理**：部署在内网网关上的执行程序，负责实际执行设备动作并上报结果。
- **目标设备**：被控制的内网主机或网络设备。
- **命令**：一次可追踪的控制请求，包含唯一标识、类型、超时/过期与参数。
- **命令终态**：命令生命周期最终结果，包含 `SUCCEEDED`、`FAILED`、`TIMED_OUT`、`CANCELLED`。
- **设备稳定标识**：用于路由与持久化的设备唯一键（以网卡 MAC 语义为准）。
- **展示标识**：面向界面显示的人类可读设备名，不作为路由与存储主键。
- **状态采集**：网关代理按周期采集设备运行指标并上报。
- **观察态**：前端当前选中的设备状态观察会话，要求切换即时生效且无旧数据残留。

## 4. 功能需求

### 4.1 设备接入与在线管理

- 系统应当支持网关执行代理主动连接控制服务并完成注册。
  - 输入：设备稳定标识、展示标识、能力声明、版本信息。
  - 处理目标：建立设备会话并更新在线状态。
  - 输出：注册确认与可查询的设备运行状态。
- 系统应当支持设备断线检测与离线回报。

### 4.2 统一控制命令提交

- 系统应当提供统一控制命令入口。
  - 输入：目标设备、命令类型、命令参数、可选超时/重试设置。
  - 处理目标：创建命令并路由到目标设备会话或重试队列。
  - 输出：该次请求的执行状态（默认等待终态，超时返回当前状态）。
- 系统应当支持的首批命令能力包括：开机、重启、关机、状态采集开关、参数读取、参数保存。
- 系统可以支持单次状态读取命令用于诊断，但不应作为前端默认状态链路。

### 4.3 命令查询与追踪

- 系统应当提供单命令查询与命令列表查询。
  - 输入：命令标识、设备过滤、状态过滤、类型过滤、分页游标。
  - 处理目标：返回命令主体与状态事件时间线。
  - 输出：可分页、可过滤、可追踪的命令视图。

### 4.4 状态推送与前端消费

- 系统应当以实时推送承担状态分发，不依赖前端轮询作为主路径。
- 系统应当提供两类状态推送语义：
  - 全量状态事件流：广播关键设备事件与快照。
  - 单设备观察流：仅推送当前被观察设备的最新状态。
- 系统应当保证切换观察目标时，新目标状态立即生效，旧目标状态不残留。
- 状态推送通道对前端应为只读，不接收业务控制消息。

### 4.5 参数管理

- 系统应当支持按设备维度读取和保存控制参数。
  - 输入：设备稳定标识、参数对象（WOL 与远程控制参数）。
  - 处理目标：参数在控制服务持久化，并同步下发执行代理。
  - 输出：参数当前值与执行状态。
- 执行代理接收参数后应在运行内存生效。

### 4.6 审计与访问行为记录

- 系统应当记录关键控制操作审计信息。
  - 输入：操作者标识、来源信息、操作类型、目标资源、结果摘要。
  - 输出：可检索、可分页的审计记录。

### 4.7 前端可感知交互要求

- 系统应当在界面上分离展示“目标状态”与“执行结果”。
- 执行结果应当可读化展示，不应直接向用户暴露原始 JSON 字符串。
- 状态采集应当提供单一开关语义（开/关/未知）并能明确区分状态。

## 5. 关键业务规则

### 5.1 角色边界

- 控制服务必须负责：命令编排、会话管理、状态机、查询、审计、参数持久化。
- 网关执行代理必须负责：设备动作执行、命令回执、状态上报。
- 控制服务不应直接执行设备动作。

### 5.2 命令状态机

- 命令生命周期应当遵循：
  - `CREATED -> DISPATCHED -> ACKED -> RUNNING -> 终态`
- 同一命令仅首个终态结果生效；重复终态回报不应覆盖既有终态，并应留下可审计记录。

### 5.3 标识与路由规则

- 设备稳定标识必须用于路由、参数存取与持久化键。
- 展示标识仅用于界面与审计可读性，不应作为路由或持久化主键。

### 5.4 心跳与状态采集规则

- 执行代理必须持续发送心跳，且不受状态采集开关影响。
- 仅在状态采集开启时，心跳中携带完整状态采样数据。
- 心跳周期与状态采样周期必须独立配置，互不绑定。

### 5.5 控制结果回传规则

- 通过 HTTP 发起的控制请求，其状态与结果必须可由该次 HTTP 响应直接获得（默认等待终态）。
- 当等待窗口超时时，系统应返回当前状态，并允许后续查询终态。
- 状态推送不应作为 HTTP 控制请求结果的唯一通道。

### 5.6 离线与重试规则

- 目标设备离线时，系统可以将可重试命令转入重试队列。
- 重试应遵循上限与退避策略，达到上限后必须给出明确失败终态。
- 控制服务重启后，不应遗留无终态悬空命令；需给出可解释的恢复结果。

## 6. 边界与异常

- 对非法输入（类型错误、缺失关键字段、越界值），系统必须拒绝并返回可读错误原因。
- 对空数据场景（无设备、无参数、无查询结果），系统应返回明确空结果语义，不应返回不确定状态。
- 对重复提交/重复回包，系统应按幂等规则处理，不应破坏既有终态。
- 对外部依赖失败（设备离线、网络中断、认证入口异常），系统应返回可诊断错误并保留审计线索。
- 对超时与中断：
  - 执行超时应进入超时终态。
  - 请求等待超时应返回当前状态并支持后续查询。
- 对通道协议不匹配，系统应显式拒绝并给出可识别错误码。

## 7. 非功能需求与约束

### 7.1 性能与稳定性

- 系统应当支持并发控制请求下的稳定运行，并具备请求限流能力（可配置）。
- 设备连接应具备断线重连能力。
- 状态推送应满足长连接持续运行需求。

### 7.2 安全

- 公网访问必须经过统一身份认证入口。
- 认证必须叠加站点允许名单，不在允许名单中的账号不得访问。
- 系统不应采用 token 持久化与 token 核心鉴权作为当前主机制。
- 设备公网链路必须使用加密通道。

### 7.3 审计

- 系统必须记录关键操作的操作者、时间、对象、动作与结果摘要。
- 审计与命令记录应支持检索与分页查询。

### 7.4 兼容性与部署环境

- 系统应适配“公网控制服务 + 内网网关执行代理”的分离部署模式。
- 认证入口与控制服务可分层部署，控制服务可仅处理内网转发流量。
- 网关侧必须满足可持续运行与自动拉起需求。

### 7.5 数据生命周期

- 控制参数应在控制服务侧持久化；执行代理仅保留运行态内存副本。
- 执行代理不应长期写入业务文件（参数、命令状态、业务日志）。
- 当前个人单租户阶段允许控制服务持有设备明文密码以保证可用性。

## 8. 明确不作约束的内容

以下内容属于实现层选择，不应视为需求约束：

- 编程语言、框架、第三方库。
- 代码目录结构与模块拆分方式。
- 类、函数、内部接口与数据库表字段命名。
- 当前实现中的线程模型、队列结构、缓存细节。
- 当前实现中的具体构建工具链与打包脚本组织方式。
- 为适配历史实现形成的临时兼容逻辑与重构过渡做法。

## 9. 待确认事项

- 现有控制接口的“路径命名是否必须冻结”为长期契约，还是仅需保证语义兼容。
- 审计日志、命令记录、设备状态数据的最短留存期与清理策略。
- 命令重试策略的全局上限（例如最大重试次数、最大等待时间）是否需要产品级统一硬限制。
- 多站点/多设备规模目标与容量指标（设备数、并发命令量、状态推送频率上限）。
- 明文设备密码策略的退出条件与替代里程碑（何时切换到更高安全等级方案）。
- 是否需要将 IPv6 作为正式支持范围与验收项。
